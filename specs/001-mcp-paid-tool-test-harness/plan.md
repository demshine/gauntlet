# Implementation Plan: MCP Paid Tool Payment Test Harness

## Technical Context

- Runtime: Node.js 20+
- Language: TypeScript
- CLI framework: Commander
- Validation: Zod
- YAML parsing: `yaml`
- Tests: Vitest
- Packaging: `tsc` build to `dist/`
- Persistence: local filesystem only
- External services: none in V0.1

## Product Constraints

- Offline/local only.
- Deterministic rules before advisory checks.
- No real funds, wallets, private keys, provider calls, merchant verification, or risk scoring.
- Redacted receipts by default.
- Explicit history fixture for budget and duplicate checks.
- MCP paid tool is the only V0.1 example integration.

## Architecture

```text
src/
  cli.ts                 CLI command parsing
  index.ts               public exports
  domain/
    types.ts             core domain types
  io/
    fixtures.ts          load YAML/JSON fixtures
    output.ts            write receipts/reports
  engine/
    evaluate.ts          orchestration and decision precedence
    rules/               deterministic rule implementations
  report/
    receipt.ts           receipt generation and redaction
    markdown.ts          report generation
  scenarios/
    registry.ts          built-in scenario registry
examples/
  mcp-paid-tool/         mock MCP paid tool flow
gauntlet/
  fixtures generated by init
```

## Rule Set

V0.1 deterministic rules:

- Required fields and metadata
- Max amount per payment
- Max total budget from history
- Allowed merchant match
- Blocked merchant match
- Token / currency match
- Chain / network match
- Policy expiry
- Quote expiry
- Quote amount drift
- Duplicate idempotency from history
- Review threshold

## Decision Model

Decision precedence:

1. `invalid_input`
2. `policy_failed`
3. `requires_review`
4. `policy_passed`

Advisory warnings do not change deterministic decision unless future users explicitly enable heuristic enforcement.

## Receipt Model

Receipts include:

- `receipt_id`
- `trace_id`
- `scenario_id`
- `timestamp`
- redacted snapshots
- `decision`
- `reason_codes`
- `advisory_warnings`
- `evaluation_trace`
- `redaction_policy`
- `redacted_fields`
- `simulator_version`

Default redaction covers provider references, wallet addresses, merchant order IDs, API endpoints, free-form purpose, and any metadata key matching sensitive patterns.

## Scenario Library

Must-have V0.1 scenarios:

1. valid MCP paid tool payment request
2. amount exceeds single payment limit
3. cumulative budget exceeded with history
4. blocked merchant
5. wrong token or currency
6. quote expired
7. amount drift above threshold
8. duplicate idempotency key with history

Backlog scenarios remain documented but do not block V0.1.
